## 致命发现：Plan B 的背景梯度压制

### 核心逻辑

Plan B per-class balanced loss 在 marker loss 中将背景视为第 5 类：
```
loss = (car_loss + truck_loss + bus_loss + trailer_loss + bg_loss) / 5
```

背景获得 20% 梯度。但背景占 75% 的 slot。
结果：模型越训练越不会说"这里没车"。

### 数值证据

| iter | false_alarm_rate | 趋势 |
|------|-----------------|------|
| 500  | 0.194 | baseline |
| 1500 | 0.308 | +59% |
| 5000 | 0.317 | +63% |

false_alarm 在整个 Plan B 训练中**单调上升**。研究员从未在报告中将此标记为 ALERT。

### 为什么研究员没发现

1. 他的注意力完全集中在 recall 上（特别是 truck_recall）
2. false_alarm 的上升被 recall 的改善所"补偿"，表面上看指标在好转
3. 精度问题被错误归因为"推理端问题"，导致方向完全跑偏

### 如果不修复

继续在 Plan B 框架上优化只会加剧背景压制。任何后续实验（调 LR、加数据增强、换 backbone）都无法解决这个**损失函数设计缺陷**。

### 修复建议（研究员不应直接采纳，需先做消融验证）

在 per-class balance 中给背景更大的权重份额：
```python
# 当前: bg 占 1/(n_fg+1) = 20%
# 建议: bg 占 50%, 前景类平分另外 50%
bg_weight_factor = n_cls_active  # 让 bg 的贡献等于所有前景类之和
loss_marker = (fg_total + bg_loss * bg_weight_factor) / (n_cls_active + bg_weight_factor)
```

但这只是假设。**必须先做消融实验验证**。

## 次要发现：theta_fine 周期性 bug 的量化影响

theta_fine 只有 10 个 bin (0-9 度)。错误的周期性处理导致：
- 预测 0, 目标 9: loss = 0.1 (应为 0.9) -> 梯度被压低 9 倍
- 预测 4, 目标 5: loss = 0.1 (正确 0.1) -> 无影响
- 影响主要集中在 fine 角度的极端误差，可能解释为什么 offset_theta 在 0.20 附近徘徊无法突破

## 研究员下一步应该做什么（但他大概率不会做）

1. 修复 theta_fine loss（2 行代码）
2. 在 per-class balance 中增加背景权重份额
3. 对以上两个修复分别做消融实验
4. 然后才能继续优化其他方面
