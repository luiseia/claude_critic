## Part 1: CEO 质询回应 —— 背景误报降了，Truck 为何跌了？

### 数值证据 (Plan C 完整轨迹)

| iter | bg_FA | truck_R | car_R | bus_R | trailer_R |
|------|-------|---------|-------|-------|-----------|
| 500  | 0.156 | 0.176   | 0.645 | 0.322 | 0.978     |
| 1000 | 0.171 | 0.114   | 0.697 | 0.609 | 0.733     |
| 1500 | 0.206 | 0.050   | 0.598 | 0.476 | 0.822     |
| 2000 | 0.294 | 0.181   | 0.542 | 0.780 | 0.978     |

### 根因：梯度预算的三重挤压

Plan C 的梯度预算分配：

**Marker Loss (cls_loss_weight=1.0):**
- bg_balance_weight=3.0 → 背景占 3/(4+3) = 42.8% 的 marker 梯度
- 每个前景类仅占 1/(4+3) = **14.3%** 的 marker 梯度

**Cls Loss (cls_loss_weight=1.0):**
- 无 bg_balance_weight（代码 line 871-881 确认，cls loss 循环仅遍历前景类）
- 4 个前景类各占 25%
- 但背景 slot 的 cls_weight=0（目标分配中 END marker 的 cls 位置权重为 0）
- 所以 cls head 完全没有"这里是背景"的学习信号

**Regression Loss (reg_loss_weight=2.0):**
- 占总 loss 的 2.0/(1.0+2.0) = **66.7%**
- 仅在正样本（NEAR/MID/FAR 标记的 slot）上计算

**最终效果**: 每个前景类在总梯度中的占比：
- Marker 贡献: 14.3% × (1.0/3.0) = 4.8%
- Cls 贡献: 25% × (1.0/3.0) × (1-bg_ratio) ≈ 2%
- Reg 贡献: ~25% × (2.0/3.0) ≈ 16.7% (但只优化空间位置，不帮助分类)
- **分类有效梯度: ~6.8% per class**

Truck 只占 GT 样本的 31%。在 6.8% 总分类梯度中 truck 的实际贡献 ≈ 6.8% × 31% = 2.1%。而背景在 marker loss 中拿走了 42.8% × 33.3% = 14.3%。

**背景的分类梯度是 truck 的 7 倍。**

这就是"特征挤压"的本质：共享表征在被强力推向"识别背景"的同时，区分前景类别（尤其是 truck vs car）的能力被挤压殆尽。

## Part 2: cls loss 中 bg_balance_weight 的致命缺失

### 新发现 BUG-8: bg_balance_weight 仅修复了 marker loss，cls loss 被遗漏

**文件**: `git_occ_head.py:871-881`

代码确认：cls loss 的 per-class balanced 分支中，**没有使用 bg_balance_weight**。

```python
# Marker loss: 有 bg_balance_weight
loss_marker += self.bg_balance_weight * bg_loss  # line 849

# Cls loss: 没有 bg_balance_weight
for c in range(self.num_classes):  # 只遍历前景类
    loss_cls += fg_loss_c
loss_cls /= n_cls_active2  # 没有背景项
```

这意味着 marker head 和 cls head 的梯度信号是**不对称**的：
- Marker head: 被强力训练去区分 object/no-object (bg_weight=3.0)
- Cls head: 只被训练去区分前景类之间的差异，对背景完全无感

**后果**：marker 可能正确预测 END (无物体)，但 cls head 在该位置仍然输出 car/truck/bus/trailer 的高置信度。虽然评估器只看 marker 为 NEAR/MID/FAR 的 slot，但自回归模型中 cls token (位置 1) 的预测会影响后续所有 token (位置 2-9)。错误的 cls 信号会通过 teacher forcing 的右移操作污染下一个 slot 的输入。

## Part 3: 30-Token 自回归结构的本质缺陷

### 发散性审计 #1: 误差级联的定量分析

30 个 token = 3 slots × 10 tokens/slot。自回归解码意味着：

```
Slot 0: [marker₀, cls₀, gx₀, gy₀, dx₀, dy₀, w₀, h₀, θg₀, θf₀]
Slot 1: [marker₁, cls₁, gx₁, gy₁, dx₁, dy₁, w₁, h₁, θg₁, θf₁]  ← 依赖 slot 0 的全部 10 个 token
Slot 2: [marker₂, cls₂, gx₂, gy₂, dx₂, dy₂, w₂, h₂, θg₂, θf₂]  ← 依赖 slot 0+1 的全部 20 个 token
```

如果 slot 0 的 marker₀ 错误地预测为 NEAR（应为 END），那么：
1. cls₀ 被迫在错误前提下选类别 → 产生一个 FP
2. gx₀~θf₀ 的 8 个回归 token 也全部产生错误梯度
3. slot 1 的 marker₁ 以 slot 0 的全部 10 个错误 token 为上下文 → 错误级联
4. slot 2 更甚

**这就是为什么 precision 在所有方案中都很难突破 0.20**。错误不是独立事件，而是在 30 个 token 中像多米诺骨牌一样倒下。

### 量化影响

假设 marker 在单个 slot 上的准确率为 p=0.85 (bg_recall ≈ 0.84)。
- Slot 0 正确概率: 0.85
- Slot 1 正确概率 (条件): 0.85 × (1 - error_propagation_factor)
- 如果 error_propagation_factor ≈ 0.1，slot 1 准确率 ≈ 0.85 × 0.9 ≈ 0.765
- Slot 2: 0.765 × 0.9 ≈ 0.69

**Slot 2 的有效 bg_recall 比 slot 0 低 19%。** 这解释了为什么 FAR 标记的物体检测效果最差。

## Part 4: Grid 分配逻辑在大尺度物体上的失效

### 发散性审计 #2: Truck/Trailer 的多 Cell 诅咒

BEV 网格: 10×10 = 100 cells, 每个 cell 10m×10m。
但目标分配使用的是 IMAGE 细网格: 20×20 = 400 cells。

物体在 IMAGE 空间中的尺寸取决于距离：
- **近处 Truck (20m 处)**: 投影到图像上约 200×100 像素 → 在 1120×1120 图像的 20×20 网格中覆盖约 **3.6×1.8 = 6-7 cells**
- **远处 Truck (60m 处)**: 覆盖约 **1.2×0.6 = 1 cell**
- **Car (20m 处)**: 覆盖约 **1.6×0.8 = 1-2 cells**

**Truck 在近处覆盖的 cell 数是 Car 的 3-5 倍。**

每个被覆盖的 cell 都会独立产生一份"正样本"标签。这意味着：
1. 一辆近处 Truck = 6-7 份独立的检测任务
2. 评估时，每个 cell 的预测独立评分
3. 如果 6 个 cell 中有 4 个正确、2 个错误 → TP=4, FP=2（同一辆车！）
4. 这辆完全被检测到的 Truck 反而拉低了 precision

更糟糕的是，IBW (Inverse-BandWidth) 权重机制给多 cell 物体分配更低的单 cell 权重：
```python
base_g_weights = {g: 1.0 / float(c) for g, c in g_counts.items()}
```
覆盖 6 个 cell 的 Truck 每个 cell 权重仅 1/6=0.167，而覆盖 1 个 cell 的远处 Car 权重为 1.0。

**效果**: 模型对近处大物体的梯度信号被人为压低 6 倍。但评估器不管 IBW — 它对每个 cell 等权评判。训练和评估的权重不一致导致近处大物体永远是"欠拟合"状态。

### 对 Truck Recall 的直接影响

Truck 比 Car 大约 2.5 倍 (12m vs 4.5m)。在所有距离上，Truck 的 cell 覆盖数平均比 Car 多 2-3 个。IBW 使 Truck 的平均训练权重仅为 Car 的 1/2~1/3。

**Truck 在训练中天然处于劣势，不是因为样本数少（31% vs Car 50%），而是因为大物体的 IBW 惩罚。**

这解释了一个所有方案都无法解决的谜团：为什么 truck_recall 始终低于 car_recall？即使 per-class balance 给了 truck 和 car 等权的类级梯度，IBW 在样本级仍然歧视 truck。

## Part 5: Plan D 预判 —— 它会失败

Plan D 的唯一改动是 reg_loss_weight 2.0→1.0。让我计算新的梯度预算：

**Plan D 梯度预算:**
- Marker loss: bg=42.8%, 每前景类=14.3%，× cls_weight/(cls_weight+reg_weight) = 50%
- 每前景类 marker 有效占比: 14.3% × 50% = 7.15%
- Cls loss: 每前景类 25% × 50% = 12.5%
- Reg loss: 50% (不直接帮助分类)
- **分类有效梯度: ~19.7% per class (vs Plan C 的 6.8%)**

改善是显著的，但根本问题仍在：
1. **IBW 对 truck 的歧视未修复** → truck recall 仍将低于 car
2. **30-token 误差级联未修复** → slot 2/3 的精度仍然差
3. **cls loss 缺少 bg_balance_weight** → marker 和 cls head 的梯度不对称仍然存在
4. **reg_loss_weight=1.0 是否足够?** Plan B 在 reg_weight=1.0 下 offset_theta=0.197 刚好达标。如果空间精度退化到 0.25，研究员可能会被迫升回 1.5

### Plan D 最可能的结果

- bg_FA: 0.15-0.20 (bg_balance_weight=3.0 在 reg_weight=1.0 下效果更持久)
- truck_R: 0.15-0.30 (比 Plan C 好，但 IBW 歧视仍在)
- car_R: 0.65-0.75 (回归 Plan B 水平)
- avg_precision: 0.10-0.15 (改善但无法突破 0.20)
- offset_theta: 0.22-0.28 (退化，可能丢失已达标的指标)

**Plan D 会在 precision 和 spatial accuracy 之间陷入不可能三角。**

## Part 6: 被所有人忽视的根本问题

### 发散性审计 #3: 323 张图够训多少参数？

nuScenes-mini 有 323 张图像。每张图像平均 ~26 个 GT 物体。
总 GT 数: 323 × 26 ≈ 8,400 个物体实例。

其中:
- Car: ~4,200 (50%)
- Truck: ~2,600 (31%)
- Bus: ~1,600 (19%)
- Trailer: ~42 (0.5%)

**Trailer 仅有 42 个实例。** 用 42 个样本训练一个 ~10M 可训练参数的模型去检测 trailer，任何看起来"好"的结果（recall=0.978）都是纯粹的过拟合假象。这个数字在统计上毫无意义。

**Bus 也只有 ~1,600 个实例。** 分散到 400 个 cell × 3 个 slot 中，每个 bus 实例平均只覆盖 3-5 个 cell。有效训练样本数 ≈ 1,600 × 4 = 6,400 个 cell-level 样本。对于一个需要同时学习 marker + cls + 8 个回归值的模型，这远远不够。

### 发散性审计 #4: 为什么没人问过"evaluation 是否公正？"

当前 slot-aligned 评估要求 (cell_id, slot_id) 严格匹配。深度排序使用相机 Z 轴（BUG-4）。

如果两个物体在同一 cell 中的深度差仅 0.01m，微小的浮点扰动就会翻转排序，导致 slot 0 和 slot 1 互换。此时预测可能完全正确（正确的物体在正确的 cell），但因为 slot 顺序不对，产生 1 TP + 1 FP + 1 FN。

**无人做过 slot-order sensitivity analysis。** 如果放松匹配条件（只匹配 cell_id，不匹配 slot_id），precision 和 recall 可能会同时大幅上升。这意味着**当前的低 precision 部分来自评估指标的过度严格，而非模型的真实缺陷。**
