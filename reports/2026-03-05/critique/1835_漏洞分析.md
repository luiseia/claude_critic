## Part 1: BUG-12 — 评估器 GT Slot 分配顺序与训练不一致 (CRITICAL)

### 发现

**训练端** (`git_occ_head.py:433`):
```python
gts.sort(key=lambda x: x['depth'])  # 按深度排序
valid_gts = gts[:NUM_SLOTS]          # slot 0 = 最近, slot 1 = 中间, slot 2 = 最远
```

**评估端** (`occ_2d_box_eval.py:88-106`):
```python
for box_info in gt_boxes_list:          # 按 gt_boxes_list 迭代顺序
    for cell_id in box_info['img_fine_cell_ids']:
        slot_idx = grid_slot_counter[cell_id]  # slot 0 = 第一个遇到的 GT
        ...
        grid_slot_counter[cell_id] += 1
```

**核心问题**: 训练按深度排序分配 slot，评估按 GT 列表的迭代顺序分配 slot。**两者的 slot 顺序不一致。**

### 具体场景

假设某个 cell 有 2 个 GT 物体:
- Object A: truck, depth=30m (在 gt_boxes_list 中排第 5 位)
- Object B: car, depth=15m (在 gt_boxes_list 中排第 12 位)

**训练时**:
- slot 0 = car (depth=15, 更近) → model 学习在 slot 0 预测 car
- slot 1 = truck (depth=30, 更远) → model 学习在 slot 1 预测 truck

**评估时**:
- slot 0 = truck (在 list 中先出现) ← **与训练相反!**
- slot 1 = car (在 list 中后出现) ← **与训练相反!**

**评估结果**:
- Slot 0: GT=truck, Pred=car → FP(car) + FN(truck)
- Slot 1: GT=car, Pred=truck → FP(truck) + FN(car)
- **0 TP, 2 FP, 2 FN** — 模型完全正确但得零分!

### 影响范围

多物体 cell 的出现频率:
- 400 cells × 3 slots = 1200 个 slot 位置
- 323 张图平均 ~26 个 GT，每个 GT 平均覆盖 ~3 cells
- 粗估: 每张图 ~78 个 (cell, gt) 对
- 假设 10% 的 cell 有 2+ 个物体 → 约 40 个 cell 受影响
- 每个受影响的 cell 产生 2 个虚假 FP → **每张图额外 ~80 个 FP**

**这解释了为什么 precision 始终无法突破 0.20:**
- 真实 FP 数 = 实际 FP + BUG-12 虚假 FP
- 如果虚假 FP 占总 FP 的 30-50%，修复后 precision 可能直接翻倍

## Part 2: BUG-12 与已知问题的关联

### 与 BUG-4 (深度排序坐标系) 的叠加

BUG-4: 训练用相机 Z 排序 (不是 BEV 距离)。
BUG-12: 评估根本不排序。

两个 bug 的叠加效应:
1. 训练的排序本身可能不完全正确 (BUG-4)
2. 评估连训练的排序都不遵循 (BUG-12)
3. **训练和评估之间的 slot 不一致是双重的**

修复优先级: BUG-12 >> BUG-4。BUG-12 是"有 vs 无"的区别，BUG-4 是"准确 vs 近似"的区别。

### 与 Precision 的直接关系

当前 all-time best precision:
- car_P = 0.143 (Plan C @500)
- truck_P = 0.086 (Plan B @500)
- avg_P ≈ 0.11

如果 BUG-12 贡献了 30-50% 的虚假 FP:
- 修复后 car_P ≈ 0.143 / (1 - 0.3 to 0.5) ≈ **0.20-0.29**
- 修复后 avg_P ≈ 0.11 / (1 - 0.3 to 0.5) ≈ **0.16-0.22**

**仅修复 BUG-12 可能就足以突破 avg_precision >= 0.20 的目标!**

## Part 3: 评估代码的其他隐患

### 隐患-1: FP 的双重计算

Line 145:
```python
frame_res['class_stats'][cls_name]['fp'] = int(c_pd_mask.sum()) - int(tp_mask.sum())
```

FP = 该类的总预测数 - TP。但这包含了两种来源:
1. 背景 cell 的误报 (真正的 FP)
2. GT cell 中类别不匹配的预测 (可能是 BUG-12 造成的虚假 FP)

当前实现无法区分这两种 FP，导致分析报告中无法判断 precision 低是因为"预测了太多假物体"还是"预测的类别/slot 对不上"。

### 隐患-2: 回归 offset 计算仅在 TP 上

Line 147-157: 只有 TP (类别和 slot 都匹配) 才计算回归误差。如果 BUG-12 导致大量 TP 变成 FP+FN，那么回归误差的统计样本量就大幅减少，offset 指标的统计意义被削弱。

### 隐患-3: 背景 FP 的统计口径

Line 130-133:
```python
bg_gt_mask = ~g_markers_mask
frame_res['bg_stats']['fp'] = int((bg_gt_mask & p_is_pos).sum())
```

这统计的是"GT 为背景但模型预测为前景"的 slot 数。但如果某个 cell 有 1 个 GT 物体 (slot 0 有 GT)，那么 slot 1 和 slot 2 会被标记为"GT 背景"。如果模型在 slot 1 预测了该物体的 duplicate (因为 30-token 自回归的误差级联)，这会被计入背景 FP，而非真正的误检。
