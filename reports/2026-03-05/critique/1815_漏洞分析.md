## Part 1: BUG-9 — 永久性梯度裁剪 (Critical)

### 发现

Plan D 配置: `clip_grad=dict(max_norm=0.5, norm_type=2)`

但训练日志（iter 10-290, 共 29 个采样点）显示：

| 统计量 | 值 |
|--------|-----|
| grad_norm 最小值 | 3.85 |
| grad_norm 最大值 | 59.55 |
| grad_norm 平均值 | 20.48 |
| >10 的比例 | 69% (20/29) |
| >20 的比例 | 41% (12/29) |
| >50 的比例 | 10% (3/29) |
| **低于裁剪阈值 0.5 的次数** | **0/29 = 0%** |

**100% 的迭代都在被梯度裁剪。** 这不是偶发现象，这是系统性的。

### 裁剪的实际效果

当 `max_norm=0.5` 而 `grad_norm=20.48` (平均值) 时：
- 实际应用的梯度 = 原始梯度 × (0.5 / 20.48) = 原始梯度 × **0.024**
- 即 96% 的梯度信息被丢弃
- 在 grad_norm=59.55 (最大值) 时: 只保留 0.5/59.55 = **0.84%** 的梯度
- 在 grad_norm=3.85 (最小值) 时: 只保留 0.5/3.85 = **13.0%** 的梯度

### 有效学习率波动

由于裁剪比例随 batch 变化，不同 iteration 的有效学习率相差：
- max/min ratio = 59.55/3.85 = **15.5x**
- 这意味着"简单" batch 的有效学习步长是"困难" batch 的 15.5 倍
- AdamW 的自适应性被梯度裁剪完全覆盖 — 模型退化为一种随机步长的 sign-SGD

### 与原始 GiT 的对比

原始 GiT 所有标准任务 (detection, segmentation, caption, grounding, instance_seg) 全部使用 `max_norm=0.1`。

研究员将 occupancy prediction 的 max_norm 从 0.1 提升到 0.5（5 倍放松），但仍然 100% 裁剪。这说明 **occupancy prediction 的梯度规模比正常 GiT 任务大 40-120 倍。**

这不是正常的。这是 loss function 设计存在严重数值问题的直接证据。

### 根因推测

梯度异常大的来源：
1. **30-token 自回归交叉熵在低概率事件上产生极大梯度**: 当模型对错误 token 赋予 >0.99 的概率时，正确 token 的 CE 梯度 ≈ -log(0.01) = 4.6，在 224 维词表上这会被放大
2. **marker loss 的 bg_balance_weight=3.0 放大器**: 背景样本约占 80%，乘以 3.0 权重后 marker loss 的梯度贡献被放大
3. **per_class_balance 导致少数类的梯度贡献被人为放大**: trailer (仅 42 个样本) 获得与 car (4200 个样本) 等权的梯度，100× 的放大倍数导致 trailer 的单样本梯度极大
4. **IBW (1/N) 权重在少量 cell 覆盖的远处物体上产生大权重**: 远处 car 覆盖 1 cell → weight=1.0，梯度不衰减

## Part 2: BUG-10 — 优化器冷启动 (High)

### 发现

Plan D 配置:
```python
load_from = 'ssd_workspace/Train/Train_20260305/plan_c_bg_fix/iter_1000.pth'
resume = False  # ← 关键
```

`load_from` + `resume=False` 的语义是：
- **加载模型权重** ✓
- **重置优化器状态** ← 问题！
- **重置学习率调度器** ← 从 iter 0 重新开始
- **重置迭代计数器** ← 从 iter 0 开始

AdamW 的核心是维护每个参数的一阶矩 (m) 和二阶矩 (v) 的指数移动平均。`resume=False` 意味着 m=0, v=0 重新开始。

### 影响

1. **Adam 偏差校正在前 100 步剧烈波动**: 当 beta2=0.999 时，v̂ₜ = vₜ/(1-0.999^t)。在 t=1 时校正因子 = 1/(1-0.999) = 1000。在 t=100 时 = 1/(1-0.905) = 10.5。这 100× 的差异导致前 100 步的有效学习率极不稳定。
2. **与永久梯度裁剪叠加**: 裁剪 + Adam 冷启动 = 双重的步长不确定性
3. **前 500 步的训练质量极差**: Plan D 的第一个验证点在 iter 500，但前 100-200 步的参数更新基本是随机游走

### Plan C 是否也有这个问题？

Plan C 从 baseline 的某个 checkpoint 恢复，使用的也是 `load_from` + `resume=False`。所以 **Plan B→C→D 的每次过渡都有优化器冷启动问题**。

## Part 3: Plan D 早期训练病理

### Loss 统计 (iter 10-290)

| 统计量 | 值 |
|--------|-----|
| 平均 loss | 1.532 |
| 标准差 | 0.615 |
| 变异系数 (CV) | **40.1%** |
| 最大/最小比 | 6.0× |

CV=40% 意味着训练过程极其嘈杂。正常的 well-behaved 模型训练 CV 应在 10-20%。

### cls/reg 比例分析

Plan D 的核心改动是 reg_loss_weight=1.0 (vs Plan C 的 2.0)。从日志看：

| iter | cls_loss | reg_loss | cls/reg 比 |
|------|----------|----------|-----------|
| 10 | 1.049 | 0.997 | 1.05 |
| 60 | 0.355 | 0.306 | 1.16 |
| 70 | 2.178 | 0.834 | **2.61** |
| 100 | 0.538 | 0.573 | 0.94 |
| 210 | 1.443 | 1.104 | 1.31 |
| 220 | 0.240 | 0.264 | 0.91 |
| 260 | 0.363 | 0.313 | 1.16 |

cls/reg 比在 0.91-2.61 之间波动。虽然平均接近 1:1 (比 Plan C 的 1:2 好)，但 **单个 batch 的比值波动高达 2.9×**。这意味着某些 batch 仍然是分类主导、另一些是回归主导，Plan D 只是在平均意义上达到了平衡，并没有解决 per-batch 的梯度竞争。

### 早期信号判读

1. **grad_norm 在下降**: 从 iter 10-30 的平均 24.7 下降到 iter 240-260 的平均 21.3，缓慢改善
2. **loss 没有明显下降趋势**: 仍在 0.5-3.0 之间震荡，没有收敛迹象
3. **极端 loss 事件频率未降**: iter 70 loss=3.01, iter 210 loss=2.55, iter 230 loss=2.32 — 高 loss 事件每 ~30 iter 出现一次

## Part 4: 梯度裁剪阈值选择的系统性错误

### 为什么 max_norm=0.5 是错误的？

标准 GiT 使用 max_norm=0.1，适配 detection/segmentation 等任务。Occupancy prediction 的 grad_norm 是标准任务的 **40-120×**，但 max_norm 只放松了 5×。

这导致两种可能：
1. **如果 max_norm 是正确的 (0.5)**：那么 loss function 的梯度规模有 40-120× 的异常，应该修复 loss function 使梯度规模回到正常范围
2. **如果 loss function 是正确的**：那么 max_norm 应该提高到 5.0-20.0 范围

研究员既没有修 loss 也没有调 clip，选择了最差的第三条路：让系统在永久裁剪状态下运行。

### 量化影响

在永久裁剪下，AdamW 退化为：
```
θ_{t+1} = θ_t - lr × max_norm × (grad / ||grad||) - wd × θ_t
```

这本质上是 **带权重衰减的 Sign-SGD**，步长固定为 `lr × max_norm = 5e-5 × 0.5 = 2.5e-5`。AdamW 的自适应学习率优势完全丧失。

但这还不是 Sign-SGD，因为梯度裁剪是对全局范数裁剪，而非逐参数裁剪。这意味着参数之间的相对梯度比例被保留，但绝对规模被压缩。结果是：梯度大的参数仍然更新更多，但所有参数的更新幅度都被压缩了 40-120×。

## Part 5: Plan D iter 500 预测 (修正版)

考虑到 BUG-9 和 BUG-10 的影响，修正 1801 报告中的预测：

| 指标 | 1801预测 | 修正预测 | 理由 |
|------|----------|----------|------|
| bg_FA | 0.15-0.20 | **0.18-0.25** | 永久裁剪削弱 bg_fix 效果 |
| truck_R | 0.15-0.30 | **0.10-0.25** | 优化器冷启动 + 裁剪 = 前500步效率低 |
| car_R | 0.65-0.75 | **0.55-0.70** | 从 Plan C iter_1000 (0.697) 出发可能退化 |
| avg_precision | 0.10-0.15 | **0.08-0.13** | 裁剪导致的训练噪声不利于精度提升 |

### Plan D 的结构性困境

即使 Plan D 在 iter 500 表现尚可，永久梯度裁剪意味着模型无法精细调整。在 iter 4000 学习率下降后 (5e-5 → 5e-6)，有效步长变为 `5e-6 × 0.5/20 = 1.25e-7`，几乎停滞。**LR schedule 的三阶段设计在永久裁剪下毫无意义。**

## Part 6: 判决

| 发现 | 新/旧 | 严重性 | 研究员意识到？ |
|------|-------|--------|---------------|
| BUG-9: 永久梯度裁剪 (100% clipping) | 新 | 致命 | 否 |
| BUG-10: 优化器冷启动 (resume=False) | 新 | 高 | 否 |
| Loss CV=40% (训练噪声) | 新 | 高 | 否 |
| cls/reg 逐 batch 波动 2.9× | 新 | 中 | 否 |
| LR schedule 在永久裁剪下失效 | 新 | 高 | 否 |
| Occupancy 梯度比标准 GiT 任务大 40-120× | 新 | 致命 | 否 |

### 给指挥家的建议

**紧急修复优先级:**
1. **诊断梯度异常**: 在 loss_by_feat_single 中对 marker_loss、cls_loss、reg_loss 分别记录 backward 前的值和 grad_norm，定位梯度爆炸的具体来源
2. **调整 clip_grad**: 要么将 max_norm 提升到 5.0-10.0 让 AdamW 正常工作，要么修复 loss function 使梯度规模回到正常范围 (grad_norm < 1.0)
3. **使用 resume=True**: 如果继续从 checkpoint 恢复训练，应该恢复优化器状态

**但根本问题不变**: 即使修复 BUG-9/10，BUG-8 (cls loss 缺少 bg_balance_weight)、IBW 大物体歧视、30-token 误差级联仍然存在。Precision >= 0.20 依然是不可达的。
